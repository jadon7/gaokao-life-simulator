# 内容、AI 对接与霍兰德结果审计文档

> 目的：把当前卡片内容、DeepSeek 生成链路、霍兰德结果逻辑、可能的冗余代码候选整合成一份评审材料。
>
> 状态：仅用于讨论和决策。本文档不代表已改产品逻辑。
>
> 审计日期：2026-06-12

## 一句话结论

当前产品已经不是“静态题库”，而是“固定剧情骨架 + DeepSeek 批量生成 + 前端二次编剧 + 本地兜底卡 + 本地霍兰德画像”的混合系统。

主要问题不在于没有接大模型，而在于“大模型负责的内容边界还不够完整”：前端仍会插入固定题、重写便签、补桥接卡、推断霍兰德、兜底结果，导致体验上容易出现“对接感”和“模板感”。霍兰德部分目前更像故事内兴趣侧写，若要让用户觉得“测得准”，需要把测量层从剧情层中拆得更清楚。

## 当前链路总览

### 事实源

- Prompt 唯一事实源：`deepseek-prompt-vnext.js`
- 本地开发 API：`server.js`
- 生产 Worker API：`worker.js`
- 前端运行时与 UI 渲染：`index.html` 和 `public/index.html`
- 内容协议文档：`docs/story-state-card-protocol.md`
- 18 张题纲：`docs/18-card-golden-outline.md` / `docs/18-card-golden-outline.json`

### API 与生成方式

当前主流程实际使用：

- `POST /api/game/batch`：按 `startYear/count` 批量生成卡片。前端在填写资料阶段就预热第一批牌。
- `POST /api/game/result`：18 张结束后生成结果页。

保留但前端主流程不直接使用：

- `POST /api/game/start`
- `POST /api/game/next`

后端调用 DeepSeek 时使用：

- `response_format: json_object`
- `temperature: 0.95`
- 第一轮流式，失败后重试一次
- 两次失败后回落到 `mockResponse`
- 所有输出都会经过 `validateAnnual` / `validateBatch` / `validateResult`

## 卡片内容机制

### 生成输入

每次生成卡片时，后端会给模型：

- 玩家资料：姓名、性别、省份、分数、期待、关键词、专业
- 固定角色表：许闻笙、周越、林知夏、合作方、家里
- 当前年份或批次年份
- 对应年份的 18 张黄金题纲
- 最近 6 条历史选择
- 最近 callback seeds
- 最近 scene titles
- lifeTrack / relationshipTrack 的当前状态

关键限制：

- 只传最近 6 条历史，不传完整 18 年细节。
- 批量预生成时，未来卡不能假设玩家未来会选 A 或 B。
- 这会让批量卡天然更“泛”，因为模型不能把未来剧情完全写死。

### Prompt 约束

当前 Prompt 已经明确要求：

- 18 张不是 18 个独立问题，而是连续短剧。
- 每张牌只拍一个事故。
- 文案面向手机卡片，scene body 60-110 字，最多 130 字。
- A/B 标题 3-5 个汉字，desc 12-22 个汉字。
- 必须有喜剧锚点、人物锚点、事故锚点、历史锚点。
- 每个选项必须带隐藏 `riasec` 和 `consequence`。
- summary 只写上一年后果 + 另一条线同步近况。
- 不要写成鸡汤、阅读理解、空话。

也就是说，方向是对的；现在的问题主要是链路分工和执行稳定性。

### 前端二次编剧

前端并不是把模型结果原样展示，它还会做这些事情：

- 把模型返回结构 normalize 成 UI 卡片结构。
- 根据上一年选择生成或合成右上角便签。
- 检测重复卡；重复时用桥接卡替代。
- 在缺卡时先显示本地桥接卡，再等远程卡回来替换。
- 每次选择后只裁掉一部分未来牌，仍保留若干预生成未来牌。
- 每 4 年插入一张本地固定卡。

这解释了“内容对接感重”的体感来源：用户看到的不是一个统一编剧一次性写出的东西，而是模型、前端规则、本地题库、兜底逻辑共同拼出来的。

### 本地固定卡

前端有一套 `fixedQuestionSeeds`，并设置：

- `fixedInsertEvery = 4`
- 年份 4 / 8 / 12 / 16 会优先插入本地固定卡。

好处：

- 保底节奏稳定。
- 可避免模型生成失败时彻底断流。
- 能把关键剧情节点钉住。

代价：

- 大模型参与感被打断。
- 同一用户多玩几次时，本地固定卡容易重复。
- 如果固定卡和模型卡口吻不同，会出现“接缝”。

## 问题 1：内容对接感重、重玩重复、幽默感还可提升

### 当前成因

1. 模型只负责部分年份，4 / 8 / 12 / 16 有本地固定插卡。
2. 批量预生成不能知道未来选择，只能写“趋势型未来”，天然不够贴脸。
3. 前端会把 summary、consequence、lifeTrack、relationshipTrack 重新压缩拼接，模型写的完整语气会被二次加工。
4. 去重只在单局内做文本相似判断，没有跨局记忆。
5. 同一资料再次开局时，固定题库基于 profile/major 做确定性 shuffle，仍可能很像。
6. Prompt 要求幽默，但没有对“幽默出现位置”和“幽默密度”做结构化验收。

### 目标体验

- 至少玩两次不重复：同一用户、同一专业、同一初始资料，第二局也应出现明显不同的事故、场景、NPC、选项动作和结果称号。
- 文案像手机短剧，不像问卷题。
- 幽默来自现实荒诞和人际反差，不靠段子硬塞。
- UI 里的标题、便签、题干、按钮、结果页称号是同一套口吻。

### 推荐方向

#### A. 增加 Playthrough Seed

每一局生成一个 `playthroughId` 或 `variantSeed`，传给后端和模型。

模型输入增加：

```json
{
  "runMeta": {
    "playthroughId": "uuid",
    "playCountForThisProfile": 2,
    "variantStyle": "workplace_comedy / campus_dramedy / relationship_pressure",
    "avoidSceneSignatures": []
  }
}
```

作用：

- 同一资料第二次玩，模型知道“这是第二局”。
- 可以要求第二局换场景、换 NPC、换事故机制。
- 未来可把已出现的 scene title / event signature 存到 localStorage，作为 `avoidSceneSignatures` 传入。

#### B. 把“前端二次编剧”收敛成“前端只裁剪”

理想边界：

- 模型输出：scene title、scene body、previous note、A/B 展示文案、A/B 后果、RIASEC 分值。
- 后端校验：长度、字段、重复、RIASEC 合法性。
- 前端只做：渲染、截断保护、降级提示。

尽量避免前端重新创造剧情句子。否则同一张牌会被模型和前端分别写半句，接缝自然会重。

#### C. 固定卡从“硬插入”改成“硬约束”

保留年份 4 / 8 / 12 / 16 的关键剧情目标，但不要直接插本地完整卡。

更好的形态：

- 本地只提供 `mustHitBeat`
- 模型仍然生成完整卡
- 后端校验它是否命中该 beat

例如第 8 年：

```json
{
  "year": 8,
  "mustHitBeat": "关系线第一次正面摊牌",
  "mustAvoid": ["合照起哄", "文件失踪"],
  "scenePoolHint": ["机房门口", "雨天食堂", "项目复盘会", "高铁站"]
}
```

这样节奏仍可控，但表面内容不固定。

#### D. 给幽默加结构化规则

现在 Prompt 只说“略带幽默”，建议拆成可验收字段：

- `comedyDevice`：群聊荒诞 / 成年人错位 / 现实反差 / 自嘲 / 弹幕感
- `comedyLine`：scene body 中必须有一处轻微反差句
- `noJokeZone`：A/B 按钮不讲段子，只讲动作和代价

验收口径：

- 题干第一句讲清事故。
- 第二或第三句给喜剧反差。
- 每张最多一个梗，不连续复用同一类梗。
- 不拿专业、家庭、地域、疾病开玩笑。

#### E. 让模型参与 UI 微文案

可以让模型输出更贴 UI 的字段：

```json
{
  "uiCopy": {
    "topBadge": "第 7 年 / 18 · 24 岁",
    "previousNoteTitle": "项目压身",
    "previousNoteBody": "上一年你接下的活开始收税，许闻笙也没再替你找借口",
    "swipeLeftHint": "先保人",
    "swipeRightHint": "先保事"
  }
}
```

前端渲染就不会显得像“模型题干 + 本地 UI 话术”拼接。

## 问题 2：霍兰德职业测试反馈准确性不足

### 当前计分方式

当前有三层霍兰德逻辑：

1. 模型为每个选项输出隐藏 `riasec`。
2. 前端选择后把 `chosen.riasec` 写入 history。
3. 如果没有 `riasec`，前端用关键词和 trait delta 推断。

结果页又有两套来源：

- 后端把 history 里的 holland 分数汇总后传给模型，让模型生成 `careerPossibilities`、结果标题、名场面等。
- 前端自己根据 history 计算 RIASEC 排名、雷达图、方向文案。

这会产生一个问题：最终职业建议是“模型生成 + 前端本地映射 + 本地兜底”的混合，不是严格测评。

### 主要风险

1. `riasec` 分值由模型临场生成，缺少量表校准。
2. Prompt 要求 A/B 围绕 `riasecAxis` 拉开差异，但后端只校验字段存在，不校验“文案和分值是否真的一致”。
3. 18 张卡的各类型曝光次数未被严格控制，可能某些类型天然更容易高分。
4. 前端显示的百分比做了视觉曲线和 rank boost，不是原始占比，容易显得过度确定。
5. 职业结果的 `careerPossibilities` 由模型返回百分比，可能看起来像概率预测，但实际只是故事侧写。
6. 专业与职业出口映射不够强，容易出现“这个专业为什么导向这个职业”的解释不足。

### 推荐方向

#### A. 测量层独立于剧情层

每张卡新增测量元数据，不只靠自然语言 Prompt 约束：

```json
{
  "measurement": {
    "itemId": "Y07_E_vs_C_delivery",
    "primaryAxis": ["E", "C"],
    "secondaryAxis": ["S"],
    "difficulty": 2,
    "contextDomain": "team_delivery",
    "aScore": { "E": 4, "A": 1, "S": 1 },
    "bScore": { "C": 4, "R": 1, "I": 1 }
  }
}
```

剧情可以灵活，测量结构要稳定。

#### B. 控制 18 张的类型曝光

建议为 18 张建立配额：

- 每个 RIASEC 类型至少有 5-6 次可得分机会。
- 每个类型至少出现 2 次主轴对立。
- 不让 C/E 过度占主导。
- 关系题不自动等于 S，稳定题不自动等于 C，冲刺题不自动等于 E。

可以在题纲层加一张矩阵表：

| 年份 | 主轴 | 副轴 | A 主分 | B 主分 | 场景域 |
|---|---|---|---|---|---|
| 1 | E/C | S | E | C | 新生项目 |
| 2 | C/I | R | C | I | 展示救场 |
| 3 | S/I | A | S | I | 关系误会 |

#### C. 后端校验分值质量

新增校验规则：

- 每个选项总分建议 5-7 分。
- A/B 主分不能相同。
- 主轴之一必须是该选项最高分。
- 最高分与第二名至少差 1 分。
- 如果文案命中明显关键词，但分值完全不匹配，拒绝重生成。

#### D. 结果页不要让模型编“概率”

更稳的做法：

1. 前端或后端确定 RIASEC code、分数、置信度。
2. 后端根据 `major × RIASEC code × choice themes` 查职业候选库。
3. 把职业候选、理由、禁用职业、置信度传给模型。
4. 模型只负责把结果写得好读、幽默、可截图。

也就是说：

- 职业方向由规则库给。
- 解释文案由模型润色。
- 模型不能凭空发明百分比。

#### E. 增加“准确性解释”

结果页可以保留娱乐感，但需要更清楚：

- “这是基于 18 次选择的兴趣倾向，不是正式心理诊断。”
- “你的前三类倾向是 X/Y/Z，代表你更常在这些情境中这样做选择。”
- “专业出口建议来自：所选专业 + RIASEC 倾向 + 18 次选择主题。”

用户会觉得更可信，而不是被一个玄学称号糊过去。

## 问题 3：可能存在没用或可收敛代码

这部分只是候选清单，不建议现在直接删。原因是当前项目有生产版、本地版、pilot、mock、fallback 多套路径，贸然删除容易伤到兜底体验。

### 高优先级候选

#### 1. `server.js` 与 `worker.js` 大量重复

两份文件都有：

- DeepSeek 调用
- JSON 解析
- annual/batch/result 校验
- mockResponse
- API 降级逻辑

风险：

- 改本地忘改生产。
- Prompt 字段或校验规则发散。

建议：

- 抽出共享模块：`api-core.js`
- server 和 worker 只保留环境适配层
- 先不动功能，只做等价迁移

#### 2. `/api/game/start` 与 `/api/game/next`

前端主流程只调用：

- `/api/game/batch`
- `/api/game/result`

`start` / `next` 目前更像旧接口或调试接口。

建议：

- 如果外部没有使用，标记 deprecated。
- 先保留 1 个版本周期，再删除。
- 文档注明主流程只用 batch/result。

#### 3. Pilot 模式

前端存在：

- `?pilot=1`
- `?pilotResult=1`
- `docs/pilot-arc-6-cards.sample.json`
- `buildPilotResult`

如果现在不再做 6 张试播，需要决定：

- 继续作为演示/回归测试模式保留
- 或迁入 `dev-tools/`，避免混在主流程里

#### 4. 固定题库与大模型题纲重叠

前端有：

- `universalCards`
- 各专业 `majors[major].cards`
- `fixedQuestionSeeds`
- DeepSeek 18 张黄金题纲

这些内容池职责重叠。

建议：

- 选一个主事实源。
- 如果主流程已是 DeepSeek + outline，就把本地题库降级为 fallback。
- fallback 文案也要按最新口吻更新，避免模型失败时风格突然变老。

### 中优先级候选

#### 5. Trait 系统与 Holland 系统并行

前端有一套 `state.values`：

- explore
- discipline
- empathy
- stability
- ambition
- pressure

同时又有 RIASEC：

- R/I/A/S/E/C

trait 影响时间线、人生节点、fallback 结果；RIASEC 影响霍兰德画像。两套系统不是完全无用，但现在边界不清。

建议：

- trait 用于“人生叙事风味”
- RIASEC 用于“兴趣测量”
- 结果页明确哪些来自 trait，哪些来自 RIASEC

#### 6. 结果页标题双重归一化

后端会 normalize result title，前端也会 normalize result title。

好处是稳，坏处是模型写的标题可能被前端改到失真。

建议：

- 后端负责结构校验和修正。
- 前端只负责显示保护。
- 若前端必须兜底，结果中标记 `titleWasRewritten` 便于调试。

#### 7. 根目录与 `public/` 双份前端文件

本地读根目录：

- `index.html`
- `assets/`

生产读：

- `public/index.html`
- `public/assets/`

风险：

- 本地看着好了，线上没变。
- 或线上资源变了，本地没变。

建议：

- 加同步脚本。
- 或把根目录改成开发源，`public/` 由脚本生成。

## 推荐改造路线

### 第一阶段：只改内容协议，不大改架构

目标：最快解决“对接感”和“重玩重复”。

建议动作：

1. 给每局增加 `playthroughId` 和 `variantSeed`。
2. Prompt 增加“第二局不重复”的规则。
3. 把最近已出现场景签名传给模型。
4. 固定卡先不删，但在文案上统一最新口吻。
5. 结果页去掉“概率感太强”的表达，改成“倾向占比 / 可能方向”。

适合：想快速上线一版内容体验提升。

### 第二阶段：重做霍兰德测量层

目标：让结果页更准、更解释得通。

建议动作：

1. 给 18 张题纲补测量矩阵。
2. 后端校验 RIASEC 分值质量。
3. 建立 `专业 × RIASEC × 选择主题` 职业候选库。
4. 模型只写文案，不生成职业概率。
5. 结果页增加置信度和解释逻辑。

适合：想把“霍兰德职业测试”作为核心卖点。

### 第三阶段：技术瘦身

目标：减少未来维护成本。

建议动作：

1. 抽出 server/worker 共享 API 核心。
2. 清理或迁移 pilot。
3. 明确本地题库只作为 fallback。
4. 建立 public 同步脚本。
5. 给内容链路加少量自动化验收样例。

适合：内容协议稳定后再做。

## 建议的验收标准

### 卡片内容

- 同一 profile 连续玩两次，18 张中至少 12 张 scene title 不同。
- 同一 profile 连续玩两次，关键场景地点至少 8 个不同。
- 每张 scene body 60-110 字，最多 130 字。
- 每张只有一个主事故。
- 每张有且只有一个轻微幽默点。
- A/B 都是动作，不是价值观口号。
- A/B 后果能自然接到下一张便签。

### 霍兰德

- 每个 RIASEC 类型至少有 5 次可得分机会。
- 每个选项分值总和稳定在 5-7。
- A/B 主分必须不同。
- 18 张后结果前三类能被至少 3 个具体选择解释。
- 职业方向必须能追溯到所选专业和 RIASEC code。
- 结果页不使用“你一定会”“高概率成为”这类预测腔。

### 结果页

- 标题三段式，14-21 字优先，最多 24 字。
- 职业出口具体到岗位或职业族，不写“相关方向的人”。
- status42 24-36 字，有高光和代价。
- famousScenes 是具体片段，不是情绪总结。
- timelineBlocks 能看出 18-22、22-30、30-35 的阶段变化。

## 最小下一步

如果只选一个最有杠杆的改动，建议先做：

**把“固定插卡 + 模型批量生成”改成“固定剧情目标 + 模型生成表面内容”，并加入 `playthroughId`。**

原因：

- 能最快降低重复感。
- 能让大模型真正接管表面文案。
- 不需要立刻重写 UI。
- 不会牵动霍兰德大重构。
